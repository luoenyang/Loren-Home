# http 请求方式

  • get、post、put、delete、head、options、trace、connect

# http状态码

  • 消息（1字头）：代表请求已被接受，需要继续处理,100 继续，101 切换协议
  • 成功（2字头）：200 OK ，202 服务器已接受请求 ，204 服务器成功处理了请求
  • 重定向（3字头）：这类状态码代表需要客户端采取进一步的操作才能完成请求。301永久移动。请求的资源已被永久的移动到新URI；302临时
  • 请求错误（4字头）：403 拒绝执行 ， 404 请求失败 找不到资源
  • 服务器错误（5）：500 服务器端的源代码出现错误 ， 501 服务器不支持当前请求所需要的某个功能

# 协议

  • HTTP协议：超文本传输协议，从WWW服务器传输超文本到本地浏览器的传送协议，无状态协议。
  • 有状态：通信的双方是必须要时刻记住当前连接的状态的，因为不同的状态下能接受的命令是不同的。
  • 无状态：每一个传输请求都是相对独立的，不需要记住当前的连接状态，一个请求的失败或成功不影响另一个请求。
  • IP(网际协议)：计算机之间的通信
  • TCP(传输控制协议) : 应用程序之间的通信
  • TCP/IP 就是TCP 和 IP 两个协议在一起协同工作，有上下层次的关系。

TCP 负责应用软件（比如你的浏览器）和网络软件之间的通信。IP 负责计算机之间的通信。

# http请求构成

    1. 请求方法URI协议/方式， https:HTTP的安全版,HTTPS的安全基础是SSL
    1. 请求头 （包含：可接受的内容类型，语言，请求的域名，浏览器端浏览器型号和版本，连接状态）
    1. 请求正文（请求正文中可以包含客户提交的查询字符串信息）

# http工作过程

    1. 地址解析(封装HTTP请求数据包) -->
    1. 封装HTTP请求数据包 -->
    1. 封装成TCP包，建立TCP连接（TCP的三次握手）-->
    1. 客户机发送请求命令 -->
    1. 服务器响应 -->
    1. 服务器关闭TCP连接
# TCP的三次握手（连接）和四次挥手（断开）

一句话：客户端和服务端都需要直到各自可收发 (发送与回应)。

三次握手
目的：三次握手可以防止旧连接的数据干扰新连接，确保数据的完整性和可靠性。确保客户端和服务器之间的连接是可靠的，双方都能够正常发送和接收数据。

  • 女朋友和程序员提出约会建议，(SYN（同步序列编号）)
  • 程序员收到女朋友的建议并且同意了，然后向女朋友提出去吃火锅，(SYN-ACK（同步序列编号-确认）)
  • 女朋友收到程序员的建议，向男朋友说好啊好啊，讨论结束。(ACK（确认）)

四次挥手

必须是四次，因为第二次之后 服务端要处理未完成的数据，之后的第三次才是真正的结束告诉客户端说
ok， game over。

  • 女朋友和程序员提出分手，
    客户端发送FIN报文
  • 程序员收到信息，向女朋友说我要考虑考虑，
    服务器回复ACK报文：
    -> 服务器收到客户端的FIN报文后，会发送一个带有ACK标志位的TCP报文段作为确认，表示已经接收到客户端的关闭请求
  • 程序员一顿骚操作后（把车放钥匙甩给女友净身出户），说那就分手吧，
    服务器发送FIN报文：
    -> 服务器在确认没有数据需要发送给客户端后，会发送一个带有FIN标志位的TCP报文段
  • 女朋友收到分手消息伤心不已，删除拉黑恋情结束。
    客户端收到服务器的FIN报文后：
    -> 会发送一个带有ACK标志位的TCP报文段作为确认，表示已经接收到服务器的关闭请求

一般情况下，一旦Web服务器向浏览器发送了请求数据，它就要关闭TCP连接，然后如果浏览器或者服务器在其头信息加入了这行代码 Connection:keep-alive TCP连接在发送后将仍然保持打开状态，于是，浏览器可以继续通过相同的连接发送请求。保持连接节省了为每个请求建立新连接所需的时间，还节约了网络带宽。

# get和post的区别

  • 参数 GET把参数包含在URL中，POST通过request body传递参数
  • 缓存 GET请求会被浏览器主动cache，而POST不会，除非手动设置
  • 编码 GET请求只能进行url编码，而POST支持多种编码方式
  • 返回 GET在浏览器回退时是无害的，而POST会再次提交请求
  • 历史 GET请求参数会被完整保留在浏览器历史记录里
  • 长度 GET请求在URL中传送的参数是有长度限制的，而POST没有
  • 安全 GET比POST更不安全，因为参数直接暴露在URL上，所以不能用来传递敏感信息

# http和https的区别

  • 1.HTTP 是不安全的，而 HTTPS 是安全的
  • 2.HTTP 标准端口是80 ，而 HTTPS 的标准端口是443
  • 3.在OSI 网络模型中，HTTP工作于应用层，而HTTPS 的安全传输机制工作在传输层
  • 4.HTTP 无法加密，而HTTPS 对传输的数据进行加密，认证的网络协议，安全性高于HTTP协议。
  • 5.HTTP无需证书，而HTTPS 需要CA机构wosign的颁发的SSL证书，一般免费证书少，因而需要一定费用。

# tcp和udp的区别

  • TCP：

    • 是一种面向连接的、可靠的、基于字节流的传输层通信协议。
    • 在数据传输前需要先建立连接，通过三次握手过程来确保两端的通信是同步的。
    • 传输数据稳定可靠，适用于对网络通讯质量要求较高的场景。

  • UDP：

    • 是一种面向数据报的协议，是无连接的。
    • 在发送数据之前不需要建立连接，直接将数据包发送给接收方。
    • 不保证数据包的顺序、完整性或可靠性，因此传输速度较快，但可能产生丢包。

# 头部开销

  • TCP：
    • 首部长度较长，会有一定的开销。在没有使用“选项”字段时，首部是20个字节；如果使用了“选项”字段，则会变长。
  • UDP：
    • 首部只有8个字节，并且是固定不变的，开销较小。

# 应用场景

  • TCP：
    • 适用于需要高可靠性的应用，如网页浏览（HTTP/HTTPS通常运行在TCP之上）、电子邮件、文件传输（如FTP）等。
  • UDP：
    • 适用于对传输速度和效率要求高、可以容忍一定数据丢失的应用，如在线视频会议、实时游戏、流媒体、DNS域名查询、SNMP网络管理等。
