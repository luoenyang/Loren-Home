# 为什么存在跨域

  • 如果缺少了同源策略，浏览器很容易受到 XSS、CSFR 等攻击。
  • 所谓同源是指"协议+域名+端口"三者相同。

# 5. 跨域资源共享（CORS）

  • 服务端设置：res.writeHead("Access-Control-Allow-Origin", "*")，或者是某个允许的网站
  • 普通跨域请求：只服务端设置 Access-Control-Allow-Origin 即可，前端无须设置；
  • 若要带 cookie 请求：前后端都需要设置。
// jquery
$.ajax({
    ...
   xhrFields: {
       withCredentials: true    // 前端设置是否带cookie
   },
   crossDomain: true,           // 会让请求头中包含跨域的额外信息，但不会含cookie
    ...
});

// axios
axios.defaults.withCredentials = true



// 服务端：跨域后台设置
res.writeHead(200, {
    'Access-Control-Allow-Credentials': 'true',                       // 后端允许发送Cookie
    'Access-Control-Allow-Origin': 'http://www.demo1.com',            // 允许访问的域（协议+域名+端口）
    // 'Access-Control-Allow-Origin': "*",
    'Access-Control-Allow-Methods': "*",
    'Access-Control-Allow-Headers': "*",
    'Set-Cookie': 'l=a123456;Path=/;Domain=www.domain2.com;HttpOnly'  // HttpOnly的作用是让js无法读取cookie
});

# 6. nginx 代理跨域

  • 原理，同源策略是浏览器的安全策略，服务器不存在跨越问题。
# proxy服务器
server {
    listen       8081;
    server_name  www.domain1.com;
    location / {
        proxy_pass   http://www.domain2.com:8080;  #反向代理
        proxy_cookie_domain www.domain2.com www.domain1.com; #修改cookie里域名
        index  index.html index.htm;





        # 给接口添加支持 CORS 的配置（也可以全局配置，放在server里）：
        # 配置是为了从别的域名、端口号也可以访问nginx所代理的域名端口号，从而实现别的域名可跨域访问nginx



        # 当用webpack-dev-server等中间件代理接口访问nignx时，
        # 当前端只跨域不带cookie时，可为*
        add_header Access-Control-Allow-Origin http://www.domain1.com;  
        add_header Access-Control-Allow-Credentials true;

        # 或

        # 给OPTIONS添加200的返回，是为了处理在发送POST请求时Nginx依然拒绝访问的错误,
        # 发送"预检请求"时，需要用到方法 OPTIONS ,所以服务器需要允许该方法。
        if ($request_method = 'OPTIONS') {
            add_header Access-Control-Allow-Origin * always;
            add_header Access-Control-Allow-Methods 'GET,POST,PUT,DELETE,OPTIONS';
            add_header Access-Control-Allow-Headers 'DNT,X-CustomHeader,Keep-Alive,User-Agent,X-Requested-With,If-Modified-Since,Cache-Control,Content-Type,Content-Range,Range';
            return 200;
        }
        add_header Access-Control-Allow-Origin '*' always;
        add_header Access-Control-Allow-Methods 'GET, PUT, POST, DELETE, OPTIONS';
        add_header Access-Control-Allow-Headers 'DNT,X-CustomHeader,Keep-Alive,User-Agent,X-Requested-With,If-Modified-Since,Cache-Control,Content-Type,Content-Range,Range';
        # 如果跨域请求携带了自定义请求头，那么服务器端需要设置 Access-Control-Allow-Headers 响应头，以允许这些请求头。如果服务器端没有正确设置Access-Control-Allow-Headers，那么即使设置了Access-Control-Allow-Origin: *，仍然会出现CORS错误。

        proxy_pass  http://ip:port/thing;
    }
}

# 8. WebSocket 协议跨域，建立长连接的通信方式（前后台尽量使用原生）

  • WebSocket protocol 是 HTML5 一种新的协议。
  • 它实现了浏览器与服务器全双工通信，同时允许跨域通讯（非同源也可以通信，不受同源策略的限制）。
  • 有别于eventsource 详见eventsource和websocket的区别
websocket是如何传输数据的 是通过http吗?

WebSocket不是通过HTTP进行数据传输的。WebSocket是建立在一个持久连接上的，可以在浏览器和服务器之间直接进行双向通信。这种通信方式不需要每次发送数据时都建立新的连接，而是可以在已经建立的WebSocket连接上直接发送和接收数据。

WebSocket的建立过程包括一次握手，这个握手过程是通过HTTP协议完成的。在建立握手后，WebSocket连接就建立起来了，之后的数据传输就不再需要使用HTTP协议。

在WebSocket连接建立后，客户端和服务器可以通过这个连接直接进行数据交互，包括发送和接收数据。这种数据交互是双向的，即客户端可以向服务器发送数据，服务器也可以向客户端发送数据。这种双向通信的方式使得WebSocket在实时应用如在线聊天、实时推送等领域具有很高的灵活性。

总的来说，虽然WebSocket的建立过程涉及到HTTP协议，但其数据传输过程并不依赖于HTTP协议。

WebSocket 有跨域的问题吗?

WebSocket 可以在不同源之间进行通信，并且它不受同源策略的限制。在 Web 开发中，常规的 HTTP 请求（例如 AJAX 请求）受到同源策略的约束，限制了在浏览器中进行跨域通信的能力。但是，WebSocket 协议允许在不同源之间建立双向通信的连接，绕过了同源策略的限制。这使得使用 WebSocket 进行跨域通信成为可能，因为它不受同源策略的限制。
